  <action>
Create `internal/search/interpreter.go` in package `search`.

Define `ProductData` struct (per TASKS.md spec -- decoupled from models.Product interface):
```go
type ProductData struct {
    ID       string
    Name     string
    Brand    string
    Category string
    Price    float64
}
```

Define `Expression` interface:
```go
type Expression interface {
    Interpret(product ProductData) bool
}
```

Implement terminal expressions:
- `BrandExpression` with `Brand string` field. `Interpret` checks `strings.Contains(strings.ToLower(product.Brand), strings.ToLower(e.Brand))`.
- `PriceLessThanExpression` with `Price float64` field. `Interpret` checks `product.Price < e.Price`.
- `CategoryExpression` with `Category string` field. `Interpret` checks `strings.EqualFold(product.Category, e.Category)`.

Implement non-terminal expression:
- `AndExpression` with `Left, Right Expression` fields. `Interpret` returns `e.Left.Interpret(product) && e.Right.Interpret(product)`.

Implement `Parse(query string) (Expression, error)` function:
1. Split query by " AND " (case-sensitive separator, matching TASKS.md spec `brand:Royal AND price:<500`).
2. Parse each token: if starts with "brand:" -> BrandExpression, "price:<" -> PriceLessThanExpression (strconv.ParseFloat for the number), "category:" -> CategoryExpression. Return error for unknown format.
3. If multiple tokens, combine left-to-right with AndExpression (left-associative).
4. Add `maxDepth` constant = 32. Count AND clauses; if > maxDepth, return `fmt.Errorf("query too complex: exceeds maximum depth of %d", maxDepth)`. This guards against stack overflow per Pitfall 7.
5. Return error if query string is empty or contains no valid expressions.

Follow project conventions: Ukrainian doc comments on exported types, PascalCase exports, receiver variables use single letter (e for expression, p for product data). Use only stdlib (strings, strconv, fmt).
